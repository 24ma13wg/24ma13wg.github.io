Skills matrix

July, 2018

Self assessed

The above prototype is based on a set of assumed user stories:

  • As a resourcer, I want to know what skills we are short of so that I can recruit new people who have those skills
  • As a resourcer, I want to know what skills we are short of, and who wants to increase a certain skill, so that I can organise appropriate training
  • As a resourcer I wan to know who is available to coach/teach others a certain skill
  • As a delivery manager, I want to know who has a skill, at a level that can deliver a project, so that I can select the right people for the project
  • As a resourcer/delivery manager, I want to know who wants to increase a certain skill, so that they can be paired with more skilled people on projects
  • As a developer, I want to know what skills are strategically important to the company, and where there are deficiencies, so that I may develop my skills in areas where there is a shortage
  • As a developer, I want to know who else is learning a skill so that I might get in touch and share knowledge.

Hover over the data points to see who has the skill. The legend indicates how developed/developing their skill level is. They may also be filtered with the select list. The data points are generated from a simple, but hopefully versatile, data object:

{
  "name": "Charlie Bloggs",
  "category": "Back end",
  "skill": "Drupal",
  "project": true,
  "student": false,
  "teacher": false
}

Data capture would be handled by a separate application. I think that grading skill level, either by self assessment and/or by peer review, is problematic. Binary questions are better:

  • Do you have any x skills: yes/no
  • Do you feel confident applying skill x to a client project: yes/no
  • Is it a priority for you to increase your x skills: yes/no
  • Do you feel confident to teach/coach skill x to others: yes/no

For example, if you feel confident that you could be part of a team that delivers a client project built mainly with Vue.js, then you should register Vue.js as a project skill. If you are not yet at this level, but can build applications with Vue.js, you should register it as a skill. If you are currently studying Vue.js, mark yourself as a student. If you feel you can, and want to, teach Vue.js to others, then mark yourself as a teacher.

Some considerations...

  • How are we define a skill? Just code skills?
  • How granular can/should we get with identifying skills? For example: Back end > PHP > Drupal > Drupal 8 > Block plugins?
  • Data privacy: skills you don't have, or have not developed yet, could be considered sensitive information.
  • This does not identify project availability.

Robot assessed

This second prototype is more matrix-like. It uses commit data (number of commits), scraped from GitHub by Florian's robot, and indexed by Elasticsearch.

Please scroll right ⇒ * Only includes commits by GitHub accounts using an email address that contains the sub-string wunder.

Some considerations...

  • There is a lot of data to be displayed; both skills and developers need to be filterable, maybe by back-end/dev ops/front-end or Estonia/Finland/Germany/Latvia/UK – whatever is useful. More tags would probably fix this.
  • It relies on a skill being identified from the code of a commit. There could be benefit in inviting pull requests for skill tag creators from anyone who wants to see their skill identified and included in the matrix
  • If someone changes/uses different GitHub names, then they will appear multiple times. Could we add a usernames.txt file to Elasticsearch, so that it knows to use only a single username for indexing?
    • [
  {
    "primary": "Charlie Bloggs",
    "others": ["CBloggs", "Zelda"]
  },
  {
    "primary": "chrisb",
    "others": ["ChristopherBloggs"]
  }
]
  • Developers who have been with the company longer – and made more commits – will appear more skilled, which may, or may not, be fair. We might display data from only the past one year, by default?
  • How do we signal skills that cannot be identified by project commits? Could we have a missing.txt file, which will be scraped by the robot, and which tells the robot about other skills. This could be represented by a different colour in the table (if no commits are found).
    • [
  {
    "username": "Charlie Bloggs",
    "missing": ["React", "d3"]
  },
  {
    "username": "chrisb",
    "missing": ["Less, Sass"]
  }
]
