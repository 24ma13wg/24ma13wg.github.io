What is a skills matrix?

In the context of this project, a skills matrix lists people along one axis, and skills along another. At the intersections will be some indication as to whether or not a person possesses the skill and if they do, perhaps, to what degree.

Why build a skills matrix?

The matrices, below, aim to satisfy the following user stories:

  • As a talent manager, I want to know what skills we are short of so that I can recruit new people who have those skills
  • As a talent manager, I want to know what skills we are short of, and who wants to increase a certain skill, so that I can organise appropriate training
  • As a talent manager I want to know who is available to coach others a certain skill
  • As a talent coordinator, I want to know who has a skill, at a level that can deliver a project, so that I can select the right people for the project
  • As a talent coordinator, I want to know who wants to increase a certain skill, so that they can be paired with more skilled people on projects
  • As a developer, I want to know what skills are strategically important to the company, and where there are deficiencies, so that I may develop my skills in areas where there is a shortage
  • As a developer, I want to know who else is learning a skill so that I might get in touch and share knowledge.

Get agreement on these user stories.

Ways to generate skills data

There are three ways to capture skills data about people, by:

  • Self assessment
  • Peer assessment
  • Robot assessment

Self assessed matrix

Hover over the data points to see who has the skill. The legend indicates how developed/developing their skill level is. They may also be filtered with the select list. The data points are generated from a simple, but hopefully versatile, data object:

{
  "name": "Josiah Bloggs",
  "category": "Back end",
  "skill": "Drupal",
  "project": true,
  "student": false,
  "teacher": false
}

Data capture would be handled by a separate application. I think that grading skill level, either by self assessment and/or by peer review, is problematic. Binary questions are, perhaps, better:

  • Do you have any x skills? Yes/No
  • Do you feel confident applying skill x to a client project? Yes/No
  • Are you currently learning skill x? Yes/No
  • Do you feel confident to teach/coach skill x to others? Yes/No

For example, if a developer feels confident that they could be part of a team that delivers a client project built mainly with Vue.js, then they should register Vue.js as a project skill. If they are not yet at this level, but can build applications with Vue.js, they should register it as a skill. If they are currently studying Vue.js, they should indicate that they are a student. If they feel able, and want to, teach Vue.js to others, then they should indicate that they can be a teacher.

GitHub commits questions

  • Should skills and users be filterable? Probably yes because there is a lot of data. More tags would probably fix this. For example:
  • How can we teach Elasticsearch to identify all the different skills? The results rely on a skills being identified from the code of commits. There may be benefit in inviting PRs from anyone who can develop a good test to ensure their skill is identified and included in the matrix.
  • How can we fix the situation where someone changes/uses different GitHub names? They will appear multiple times! Could we add a usernames.txt file, so that Elasticsearch knows to use only a single username for indexing?

Review with Parsla, 23 August 2018

  • The user stories look good
  • For the self-assessed matrix, there needs to be another skill level represented, something like: I know nothing, but I'm interested in learning this skill
  • Also, for the self-assessed matrix, it needs to be more obvious who the people are in the matrix; just hovering over the dots to see a name is not sufficient: a personal profile is needed
  • There should be more ways of filtering the data in the self-assessed matrix
  • We agreed that the best place for any application would be the Intranet: a better form for capturing people's skills, and a more insightful skills matrix – with some data only visible to users with sufficient permissions
  • The table format used with the GitHib data is good – it would be good to incorporate the data from GitHub, somehow…
  • Using the data in Harvest to generate a skills matrix is perhaps not a good approach because the data is not so useful

Final application

A final application could have three interfaces:

  1. A data capture form
  2. Some filters
  3. A data visualisation

My initial thoughts about the design is that this application is, essentially, two things: (1) a big picture of the company's skills; and (2) a people search.

With regard to point two, I think some ideas can be taken from an online shop. Let's say I want to buy a pair of new shoes. The starting point of my search is with every pair of shoes in the shop. I refine my search by mens, then by size, and finally let's say I want them in blue. I now have a shortlist, from which I can examine the details of the pairs I like. The design of our application could be similar, except instead of returning a list of items after every refinement, we return a matrix

For the matrix, I like sortable tables, which can be expanded and collapsed to show more, or less, detail.

The data capture form can be a simple form of fieldsets.
